# The Argent Order

## Yearly Operating Plan

Version: 1.0

Status: Canonical System Deployment Timeline

---

# Purpose

This document defines how The Argent Order is **built, launched, and scaled over time**.

It is not theoretical.

It is an execution sequence.

The system is designed in layers:

```text id="purpose"
Build → Launch → Stabilise → Scale → Multiply → Institutionalise
```

---

# CORE DESIGN RULE

You do not scale everything at once.

You scale **one formation layer at a time** until it stabilises.

---

# PHASE 0 — FOUNDATION (BUILD PHASE)

## Objective

Build the minimum viable formation system.

---

## What is built:

* Portal (core tracking system)
* Discord server (structure + pods)
* Basic Rule of Life system
* Campaign system (simple version)
* Basic gamification tracking
* Manual leadership roles

---

## Key Principle:

```text id="phase0"
No scale. Only structure.
```

---

## Exit Condition:

System can support:

* 20–50 active users
* stable daily check-ins
* functioning pods

---

# PHASE 1 — INITIAL LAUNCH (0–3 MONTHS)

## Objective

Test real human behaviour.

---

## Focus:

* onboarding flow
* pod formation stability
* discipline adherence
* basic retention

---

## Actions:

* invite first cohort
* manually assign pods
* run weekly pod calls
* collect feedback constantly

---

## System Priority:

```text id="phase1"
Retention > Growth
```

---

## Success Indicators:

* > 60% weekly retention
* functioning pods without founder intervention
* consistent campaign completion

---

## Failure Indicators:

* pod collapse rate >30%
* unclear onboarding
* inconsistent participation

---

# PHASE 2 — STRUCTURAL STABILISATION (3–6 MONTHS)

## Objective

Remove founder dependency from daily operations.

---

## Focus:

* leadership formation (Captains)
* automation of tracking
* structured onboarding flow
* standardised pod calls

---

## Actions:

* promote first Captains
* define leadership pipeline
* formalise campaigns
* introduce analytics tracking system

---

## System Shift:

```text id="phase2"
Manual system → Structured system
```

---

## Success Indicators:

* pods self-sustaining
* leaders running calls without founder
* reduced manual intervention

---

# SCALING LAW

Scaling is only allowed when:

```text id="scaling_law"
Current layer is stable AND leadership exists for next layer
```

No premature scaling.

---

# FAILURE MODES BY PHASE

---

## Phase 0–1 Failure

* overbuilding features
* weak onboarding
* unclear structure

Fix: simplify system

---

## Phase 2 Failure

* weak leadership
* founder dependency remains

Fix: aggressively delegate

---

## Phase 3 Failure

* growth without retention
* content without conversion

Fix: tighten onboarding + pods

---

## Phase 4 Failure

* leadership inconsistency
* regional drift

Fix: standardise doctrine

---

## Phase 5 Failure

* events become entertainment
* physical dependency

Fix: re-anchor digital system

---

## Phase 6 Failure

* bureaucracy
* loss of formation intensity

Fix: reinforce discipline culture

# FINAL PRINCIPLE

The Argent Order is not launched as a product.

It is deployed in phases like an operating system.

Each phase must be stable before the next is activated.

Speed without stability destroys formation.

Stability without scale preserves it.

The goal is not rapid growth.

The goal is durable transformation at scale.
